## Section Copy

### Social Proof Bar
200+ engineering teams. 1 billion API calls per month. Featured in TechCrunch.

### Origin Story

Every API integration we shipped followed the same pattern: two days reading docs, three days guessing at edge cases, and a week debugging responses that didn't match the documentation.

We kept asking the same question: why is integrating an API so painful when the API itself works fine?

The answer was always the same — the documentation was written by the people who built the API, for the people who built the API. Not for the people trying to use it.

So in 2021, we quit our jobs and built CodeBridge.

### What We Believe

API documentation shouldn't require a computer science degree to understand. It should work like a conversation — you ask a question, you get a clear answer, and you move on.

We believe integration timelines should be measured in hours, not sprints. That error messages should tell you what went wrong and how to fix it, not just what HTTP code you got back. That every developer deserves working examples they can copy, paste, and adapt — not pseudocode that looks good in a README.

### What CodeBridge Does

CodeBridge sits between your application and the APIs you depend on. We make integration painless by:

- **Generating production-ready integration code** from API specs — not boilerplate, actual working code your team can ship
- **Validating every request and response** in real time so you catch mismatches before they reach production
- **Providing interactive docs that update automatically** when an API changes — no more chasing stale documentation
- **Running 1 billion API calls per month** across our infrastructure, so you know this works at scale before you bet your product on it

### How We're Different

Most API tooling is built by companies that also sell APIs. We don't have an API to protect. Our only job is to make the APIs you already use easier to integrate.

That means our docs are honest about limitations. Our error messages explain what actually happened. And our integration timelines are based on what we've seen work — not what sounds good on a landing page.

### The Team

Three engineers. No MBAs. No "Head of Synergy."

We've collectively shipped integrations for 50+ APIs across fintech, healthcare, logistics, and developer tools. We've felt every pain point we solve — because we lived them first.

### By the Numbers

- **200+** business customers — from two-person startups to public companies
- **1 billion** API calls processed per month
- **73%** average reduction in integration time
- **Featured in** TechCrunch, Hacker News, and Dev.to

### Final CTA

You've got APIs to integrate and a roadmap that doesn't have room for six-week integration sprints. Let CodeBridge handle the documentation so your team can focus on the product.

## Annotations

**Headline — "We got tired of reading bad API docs so you don't have to."**
Uses the "We experienced the same pain" framing. Peer-level, not preachy. The "so you don't have to" creates a direct benefit for the reader without overselling. Matches the technical-but-warm voice — this sounds like something a senior dev would actually say.

**Subheadline — "Three engineers who spent years wrestling..."**
Specific origin detail builds credibility. "Wrestling" is active and visceral — not corporate. "The tool we wished existed" is honest positioning — we're not claiming to reinvent anything, we're admitting we built what we needed.

**CTA — "See CodeBridge in Action"**
For an About page, the CTA should drive product exploration, not a hard sell. "See...in Action" is lower commitment than "Start Free Trial" — appropriate for visitors who are still learning who you are.

**Social Proof Bar — "200+ engineering teams. 1 billion API calls per month. Featured in TechCrunch."**
Three specific proof points in one line. The numbers are concrete (not "many" or "thousands"). TechCrunch mention anchors credibility without overclaiming — it says "featured," not "award-winning."

**Origin Story — "Every API integration we shipped followed the same pattern..."**
Opens with a shared experience every developer recognizes. No hero narrative — just a problem that frustrated us enough to do something about it. The line "the documentation was written by the people who built the API, for the people who built the API" names the root cause without blaming anyone.

**What We Believe section**
Uses short, declarative sentences. Each belief maps to a specific pain point: bad docs, long timelines, unhelpful errors, missing examples. This section builds emotional alignment with the reader before the product pitch.

**Benefits section — "Generating production-ready integration code..."**
One benefit per bullet. Each follows the pattern: [what it does] — [why it matters]. The parenthetical "not boilerplate, actual working code your team can ship" uses contrast to reinforce the value. No jargon that would confuse the target audience.

**How We're Different — "We don't have an API to protect"**
This is the key differentiator — and it's stated in one sentence. The three short follow-up sentences ("our docs are honest... our error messages explain... our integration timelines are based on...") create rhythm and reinforce the contrast with competitors.

**Team section — "Three engineers. No MBAs. No 'Head of Synergy.'"**
Self-deprecating humor matches the voice. Signals this is a technical company that doesn't take itself too seriously. The brevity is the point — no long bios that nobody reads.

**By the Numbers — "73% average reduction in integration time"**
Specific percentage is more credible than "dramatically faster." Placed after the product explanation so the number has context.

**Final CTA — "Let CodeBridge handle the documentation so your team can focus on the product."**
Restates the core value proposition in one sentence. The three CTA options (trial, docs, demo) accommodate different visitor intents — from "just let me try it" to "let me evaluate this properly."

## Alternatives

### Headline Options

**Option A:** We got tired of reading bad API docs so you don't have to. — Peer-level, story-driven. Works because it immediately establishes shared experience with the audience.

**Option B:** Bad API docs cost you weeks. We built the fix. — More direct, benefit-focused. "Weeks" is specific and painful. "We built the fix" is confident without being arrogant.

**Option C:** Your last API integration took too long. Ours did too — until we built CodeBridge. — Conversational and empathetic. The dash creates a pivot from problem to solution. Works for an audience that appreciates honesty over polish.

### CTA Options

**Option A:** See CodeBridge in Action — Low-commitment, appropriate for About page visitors still learning about you.

**Option B:** Start Free Trial — Standard, action-oriented. Use if the About page has strong conversion intent.

**Option C:** Read the Docs — For the developer audience, reading docs is a natural next step. Low-pressure and respects their evaluation process.

### Subheadline Options

**Option A:** Three engineers who spent years wrestling with incomplete documentation, cryptic error messages, and integration timelines that blew past every estimate. So we built the tool we wished existed. — Full origin story with emotional weight. "Blew past every estimate" names a real pain.

**Option B:** We spent a combined 15 years integrating other people's APIs. The documentation was always the bottleneck. So we fixed it. — Shorter. "The bottleneck" is precise language that resonates with technical leaders.

**Option C:** 200+ teams trust us to make their API integrations painless. Here's why we started. — Leads with social proof instead of origin. Works when the About page needs to establish credibility before the story.
